Running the runner

The runner is a single zero-dependency file. You need Node.js 18+ — nothing else. Start it in a terminal:

node runner.mjs                 # root = the folder you run it from
node runner.mjs <root-folder>    # root = a folder you choose

The path argument is optional. It sets the root — the boundary the folder browser stays within — not necessarily the project itself. Omit it and the folder you launched from becomes the root; then pick the actual working folder inside the root from the control panel. Pass a project folder directly to start right there, or a parent folder to browse more widely — pass / on macOS/Linux, or a drive root such as C:\ on Windows, to browse the whole machine.

It prints a URL and a token; paste those into the app (Build mode → Local runner) to connect. Leave the token empty in the app to build fully in-browser instead.

Flags

  • --port <n> — listen on a specific port (default 8787).
  • --token <secret> — use a fixed token instead of a random one.
  • --mcp "<name>=<command>" — bridge a stdio MCP server (repeatable). You can also add these live in the control panel.
  • --context7 — shortcut to bridge Context7 docs (optional --context7-key <key>).
  • --searxng --searxng-url <url> — bridge web search via your SearXNG instance.
  • --host <addr> — expose beyond loopback (see Remote access). Refuses to start without a strong --token.

The control panel

Open http://127.0.0.1:8787/ in a browser (or click Open control panel in the app once connected). From the panel you can:

  • Browse and pick the project folder (confined to the folder the runner was started in, or an explicit root you pass as the first argument).
  • Watch live activity — the same log that prints in your terminal, streamed over SSE.
  • See your MCP servers' status and tool counts, toggle them on/off, and add or remove them at runtime.
  • Apply a verified self-update when one is available.

Folder & root

The first argument is the root — the boundary of the folder browser. If you omit it, the folder the runner was launched from is used. The browser can pick any subfolder within the root as the active working folder; it can never escape above the root. To browse the whole machine, pass an explicit root such as / (or a drive root on Windows).

MCP servers

An MCP server bridges external tools to the AI team — a real browser via Playwright, current library docs via Context7, web search via SearXNG, or anything that speaks stdio MCP. Add one in the panel with a name and the command that launches it, e.g. npx @playwright/mcp@latest. Adding an MCP server runs an arbitrary command, so only add ones you trust.

Two built-in shortcuts save you the command: --context7 bridges Context7 docs (optional --context7-key <key> or CONTEXT7_API_KEY for higher rate limits), and --searxng --searxng-url <url> bridges web search through your SearXNG instance. The first time a server launches via npx, it pauses briefly while the package downloads.

Remote access (advanced)

By default the runner binds 127.0.0.1 and is reachable only from your machine — this is the safe default and needs no extra setup. If you want the hosted app to drive a build on another machine, you can expose the runner, but understand the trade-off: it grants file and shell access, so the token is the only thing protecting it.

Because the app is served over https, it can only reach a runner over a browser-trusted https URL. The simplest way to get one is a tunnel — keep the runner on plain http and let the tunnel provide https:

  1. ngrok (best for the live log stream): ngrok config add-authtoken <token> then ngrok http 8787. The free plan gives a stable *.ngrok-free.app URL. Paste that https URL + the runner token into the app.
  2. Tailscale Funnel: tailscale funnel 8787 → a stable *.ts.net URL (public listener is limited to ports 443/8443/10000).
  3. Cloudflare named tunnel (requires your own domain) — and verify the live-log panel actually streams. Do not use a Cloudflare Quick Tunnel (trycloudflare): it buffers server-sent events, which freezes the live log.

Start the runner with --host 0.0.0.0 --token <a long random secret> and point your tunnel at its port. Reaching a runner by a bare LAN IP (http://192.168.x.x) is discouraged: browsers block it as mixed content from an https page, and Chrome additionally gates local-network access behind a permission prompt — a tunnel avoids both.

Self-update

When a newer runner is published, the panel shows an Update button. Applying it downloads the new runner, verifies its Ed25519 signature and SHA-256 against the pinned key (so a compromised download host alone cannot push malicious code), swaps the file (keeping a .bak), and restarts on the same port and token. Updates are never automatic — they only happen when you click.
