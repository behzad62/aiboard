# Benchmark feature review

_Generated 2026-07-01 by a multi-agent review (96 agents across 10 subsystems; every finding re-checked by an adversarial verifier that re-read the cited code). Findings below are the **verifier-confirmed** set; rejected candidates are listed at the end._

**Scope:** the `/benchmark` page and everything behind it — `lib/benchmark/**`, `lib/client/model-stats.ts`, and `components/benchmark/**` (~20k lines + 45 components, 9 tabs).

> **UI caveat:** live browser rendering was unavailable during the review (the automation browser was locked), so UI findings are from component source, not screenshots.

**The dominant theme is scoring validity** — the benchmark's whole purpose is to rank models honestly, and most high-severity findings let models earn unearned credit, get unfairly penalized, or game the verifier. Secondary themes: one export-privacy gap, and some import-robustness and UI/accessibility issues. The underlying engineering is solid (see per-subsystem strengths at the end).

## High severity (9)

### 1. invalid_case attempts are scored against the model despite being declared 'invalid for scoring' everywhere else

- **Severity:** High _(reviewer claimed critical; verifier adjusted)_ · **Confidence:** high · **Category:** scoring-validity
- **Subsystem:** Certified run engine
- **Location:** `lib/benchmark/metrics.ts:673-686`

**Evidence**

> isScoredCertifiedAttempt switches only on provider_unavailable/invalid_harness/invalid_environment/aborted_user -> false, with `default: return true`. invalid_case falls into the default branch and is treated as scored. Meanwhile failures.ts:233-239 INVALID_STATUSES includes "invalid_case"; failures.ts:229 explains it as "...so the run is invalid for scoring"; attempt-detail.ts:506-512 classifies invalid_case as kind:"excluded"; reports.ts/isCertifiedAttemptComplete and classify-run-result.isInvalidCertifiedRun also treat it as invalid.

**Impact** — An invalid_case attempt means the case fixture, setup command, or verifier config is broken (explicitly NOT the model's fault, e.g. workbench/executor.ts:126,393 emit it for verifier_failed / case_setup_failed). Because the leaderboard/pass-rate path (buildCertifiedBenchmarkDashboardData at metrics.ts:438) filters on isScoredCertifiedAttempt, these broken-environment attempts are counted as the model failing, depressing its verifiedPassRate and verifiedQuality. The metrics summary also has no excludedCaseAttempts bucket (lines 486-497), confirming invalid_case was simply omitted from the exclusion set. The UI (attempt-detail) tells the user the attempt is excluded while the leaderboard silently scores it — a direct contradiction that corrupts the benchmark's central output.

**Verifier** — Confirmed genuine. isScoredCertifiedAttempt (lib/benchmark/metrics.ts:673-686) only switches provider_unavailable/invalid_harness/invalid_environment/aborted_user -> false, with default: return true. invalid_case is absent and falls into the default, so it is scored. This contradicts the canonical invalid set INVALID_STATUSES (lib/benchmark/failures.ts:233-239) which includes invalid_case, the explanation text at failures.ts:229 ("...so the run is invalid for scoring"), isInvalidCertifiedRun (failures.ts:288-294, derived FROM INVALID_STATUSES), reports.ts:551-564, and store.ts:856-867. invalid_case is emitted only for non-model faults: executor.ts:126 (verifier_failed when the verifier crashes) and executor.ts:393 (case_setup_failed for broken case/manifest/fixture/verifier config). Data flow is live and end-to-end: buildCertifiedBenchmarkDashboardData (metrics.ts:434) computes scoredAttempts = certifiedAttempts.filter(isScoredCertifiedAttempt) (metrics.ts:438) which feeds the leaderboard, verifiedPassRate (500-505), averageVerifiedQuality (506-508), track/role leaderboards; this is wired through useBenchmarkDashboard.ts:161 into BenchmarkLab.tsx (live UI, app/benchmark). So a broken-environment invalid_case attempt is counted as the model failing, depressing its pass rate and quality. Two extra corroborations: (a) a direct same-file self-contradiction in attempt-detail.ts — line 478 returns kind:"scored" for invalid_case via isScoredCertifiedAttempt, making the explicit case "invalid_case" -> kind:"excluded" block at 506-512 dead/unreachable, so the UI's excluded handling never runs while the leaderboard scores it; (b) test-benchmark-scoring.mts:192-239 explicitly asserts the OTHER four invalid statuses return false but has NO invalid_case case, confirming a simple omission. The summary buckets (metrics.ts:486-497) also lack an excludedCaseAttempts bucket. Not dead/legacy code per CLAUDE.md (lib/benchmark is the live certified benchmark subsystem; the dead-server list does not include it). Severity is high rather than critical: it corrupts the benchmark's central metric and is triggerable by real persisted data, but only when a verifier/case fixture is actually broken (a conditional, not constant, trigger), and this is a self-hosted client-side benchmark feature with no security/multi-user data-integrity exposure.

**Suggested fix** — In lib/benchmark/metrics.ts isScoredCertifiedAttempt, add case "invalid_case": return false; alongside the other four invalid statuses so it matches INVALID_STATUSES in failures.ts. Best fix (eliminates the duplicated list that caused this drift): derive the scored/excluded decision from the shared source of truth, e.g. return isCertifiedAttempt(attempt) && !isInvalidCertifiedRun(attempt.status), importing isInvalidCertifiedRun/INVALID_STATUSES from failures.ts. Then add an excludedCaseAttempts (status === "invalid_case") bucket to the dashboard summary at metrics.ts:486-497 for parity with the provider/harness/environment/user buckets, and add an invalid_case assertion to scripts/test-benchmark-scoring.mts:192-239 to lock in the behavior. After the fix, the previously-dead case "invalid_case" branch in attempt-detail.ts:506-512 becomes reachable, restoring the "Excluded from score (Test design)" UI label that currently never shows.

### 2. Invalid/illegal model output is replaced by a near-optimal deterministic fallback that is then scored as the model's answer

- **Severity:** High _(reviewer claimed critical; verifier adjusted)_ · **Confidence:** high · **Category:** scoring-validity
- **Subsystem:** Fireworks benchmark
- **Location:** `lib/benchmark/fireworks/certified-runner.ts:320-334, 502-537`

**Evidence**

> In callFireworksAction on parse/illegal failure: `const fallback = chooseDeterministicFireworksFallback(params.state, params.playerId); ... return { action: fallback, rawResponse: call.rawResponse, call: record };`. Then runScenarioCase does `const score = scoreFireworksScenarioAction(params.benchmarkCase, call.action);` with `call.action` being the fallback, and the assertion is `passed: score >= 0.7`. Nothing penalizes `call.call.fallbackUsed` in the score.

**Impact** — chooseDeterministicFireworksFallback is a strong heuristic: for safe_play it finds the known-playable card (knowledge color+rank set) and returns play index 0 (weight 1.0); for avoid_bad_play it finds the visible playable clue (clue_rank 1 to P2) which is exactly the expected weight-1 action. So a model that returns garbage JSON or an illegal action still earns full scenario credit, and verifiedQuality/jobSuccessScore are inflated. The only signal of breakage is a recorded failure + status, but the numeric quality the leaderboard ranks on is contaminated by the harness's own solver.

**Verifier** — Confirmed real by reading the full chain. callFireworksAction (lib/benchmark/fireworks/certified-runner.ts:478-537) on parse/illegal failure returns { action: fallback } with record.fallbackUsed=true, where fallback = chooseDeterministicFireworksFallback. runScenarioCase (line 326) then scores `scoreFireworksScenarioAction(benchmarkCase, call.action)` on that fallback action with no branch on call.call.fallbackUsed; the assertion (line 331) is passed: score>=0.7. The fallback (lib/games/fireworks/ai.ts:138-201) is a strong solver: its knowledge-loop returns play index 0 when a card is known playable, and its usefulClue branch returns a legal clue targeting a playable partner card. Cross-checking scenario constructions (lib/benchmark/fireworks/scenario-packs.ts:149-321) shows the fallback lands the exact weight-1 expected action for safe_play (knowledge[0]={color,rank} on a playable card -> {play,0}), avoid_bad_play (knowledge[0]={} but partner green 1 is cluable via clue_rank:1 = the weight-1 action, not in forbiddenActions), and similarly safe_discard, endgame_play, needed_clue, critical_discard_avoidance, and all memory scenarios. So a model returning garbage JSON or an illegal action still earns full scenario credit. That score is averaged (certified-runner.ts:203-205) into the attempt score that becomes verifiedQuality (275), jobSuccessScore (276) and efficiencyScore (277). This is live code: CertifiedRunPanel.tsx imports the fireworks suite and runs it via runCertifiedBenchmark. It is NOT dead/legacy (not in CLAUDE.md's dead-module list). Severity downgraded from critical to high: (a) statusForAttempt (786-787) still flags the attempt failed_tool_use and a BenchmarkFailure record is pushed, so a parallel categorical breakage signal exists - only the numeric quality is contaminated, not the entire result; (b) blast radius is limited to benchmark/leaderboard metric fidelity for the certified Fireworks suite when a user runs it and a model emits illegal output - it does not corrupt user data, storage, or the discussion/build engines; (c) full-game cases score by final board state, so this most directly hits scenario cases. The reviewer did not misread the data flow; the only correction is severity, not reality.

**Suggested fix** — In runScenarioCase (certified-runner.ts:326-335), when call.call.fallbackUsed is true, force the scenario score to 0 before building the assertion, e.g. `const score = call.call.fallbackUsed ? 0 : scoreFireworksScenarioAction(params.benchmarkCase, call.action);`. This both removes the harness-solver credit and makes the passed: score>=0.7 assertion correctly fail. Prefer zeroing on fallbackUsed over scoring the originally-parsed illegal action, because garbage-JSON failures have no parsed action to score and the parsed-illegal path would already get 0/0.3 anyway. Keep the deterministic fallback solely to advance full-game simulation; for full-game cases, fallback turns inflate the team score only indirectly via board state, so optionally track a fallbackRate metric and exclude/penalize attempts with high fallback usage if stricter integrity is wanted. The existing failed_tool_use status and recorded failure should remain.

### 3. Memory scenarios put full color+rank into the visible knowledge object, so no recall from clue history is required

- **Severity:** High · **Confidence:** high · **Category:** scoring-validity
- **Subsystem:** Fireworks benchmark
- **Location:** `lib/benchmark/fireworks/scenario-packs.ts:256-290`

**Evidence**

> Every memory branch calls `state.hands[0].knowledge[0] = knowledge({ color, rank, ... })`, and the view exposes `cards[i].color = knowledge.color`, `rank = knowledge.rank` (hidden-view.ts:32-43). e.g. combine_color_and_rank: `knowledge({ color, rank, history: [...] })`.

**Impact** — The memory suite is supposed to test whether a model can reconstruct a card identity from old/negative/timed clue history. But both the resolved color and rank are present directly in the knowledge object the model sees, so the answer is stated explicitly and the clueHistory is decorative. Combined with the recommendations leak, the memory suite cannot distinguish models with good memory from those that just read knowledge.color/rank.

**Verifier** — Confirmed real. In lib/benchmark/fireworks/scenario-packs.ts:256-290 every memory branch sets state.hands[0].knowledge[0] = knowledge({ color, rank, history }) and the knowledge() helper (scenario-packs.ts:398-412) stores the resolved color/rank literally. The model-facing view is built by getFireworksPlayerView, which is at lib/games/fireworks/hidden-view.ts (the finding's path lib/benchmark/fireworks/hidden-view.ts is wrong, but the file and the cited lines 32-43 are otherwise accurate). hidden-view.ts:32-50 copies knowledge.color/knowledge.rank into BOTH ownHand.cards[i].color/rank AND a nested knowledge object. This whole view is serialized into the prompt: in the live certified runner via buildFireworksPrompt (lib/games/fireworks/ai.ts:75 + 90, called from certified-runner.ts:461) and in the live GameIQ suite via initialState=getFireworksPlayerView(...) (lib/benchmark/gameiq/fireworks.ts:36). So for memory scenarios the resolved identity the model is supposed to reconstruct from clueHistory/notColors/notRanks is stated explicitly in the view; clueHistory is decorative. The leak is actually worse than described: buildRecommendations (hidden-view.ts:92-100) pushes the card index into recommendations.knownPlayableCards whenever knowledge.color && knowledge.rank are set — always true in memory scenarios — so the model is also handed the exact answer index. Both consumer paths are live (reached from components/benchmark/certified/CertifiedRunPanel.tsx and the GameIQ scenarios), not dead/legacy code. No offsetting guard exists: test-fireworks-scenarios.mts only checks legality/uniqueness/digest, never that the identity is hidden. Severity high (not critical): no crash, no user-data corruption, but it fully invalidates the memory dimension of the benchmark — a model that ignores history and reads knowledge.color/rank scores perfectly, so the suite cannot distinguish memory ability. Scoring is unaffected by redaction because ground truth lives in scenario.expectedActions/scenario.state, not in the view.

**Suggested fix** — Redact the resolved identity from the model-facing view for memory scenarios while keeping it in scenario.state for scoring. Because the answer leaks through three fields, all three must be suppressed: (1) ownHand.cards[i].color/rank, (2) ownHand.cards[i].knowledge.color/rank and ownHand.knowledge[i].color/rank, and (3) recommendations.knownPlayableCards (hidden-view.ts:92-100), which re-derives the answer index from knowledge.color&&rank. Preferred approach: add an optional flag (e.g. getFireworksPlayerView(state, playerId, { redactOwnIdentity: true })) used only by the memory benchmark mappers (scenario-packs/certified-runner for memory suite + gameiq MEMORY_STRESS/combine_color_and_rank mappers) that nulls knowledge.color/knowledge.rank in the emitted view and skips the knownPlayableCards recommendation when identity is redacted; the model must then infer identity from clueHistory + notColors + notRanks. Keep notColors/notRanks populated and consider tightening them so the deduction is uniquely determined. Scorer ground truth (scenario.expectedActions, scenario.state) is untouched, so scoring still works. Add a test asserting that for memory scenarios the serialized view contains no resolved color/rank and an empty knownPlayableCards.

### 4. Player view fed to the model includes a precomputed optimal-move recommendation list, leaking the answer

- **Severity:** High _(reviewer claimed critical; verifier adjusted)_ · **Confidence:** high · **Category:** scoring-validity
- **Subsystem:** Fireworks benchmark
- **Location:** `lib/games/fireworks/hidden-view.ts:82, 86-137`

**Evidence**

> getFireworksPlayerView returns `recommendations: buildRecommendations(state, playerId, legalActions)` containing `knownPlayableCards`, `visiblePlayableClues`, and `safeDiscards`. The GameIQ prompt serializes the whole view: gameIqScenarioPrompt does `Initial state JSON: ${JSON.stringify(scenario.initialState)}` (lib/benchmark/gameiq/certified-runner.ts:251) and initialState = getFireworksPlayerView(...) (lib/benchmark/gameiq/fireworks.ts:36-39).

**Impact** — For every GameIQ Fireworks scenario the expected best action IS the content of recommendations.knownPlayableCards / visiblePlayableClues (the same engine helpers used to author the expected action). The model is handed the solution and only needs to transcribe it, so the benchmark measures JSON copying, not Fireworks reasoning. This invalidates GameIQ Fireworks scores as a discriminating signal.

**Verifier** — Confirmed genuine after tracing the full data flow. (1) getFireworksPlayerView returns recommendations = buildRecommendations(...) (lib/games/fireworks/hidden-view.ts:82), where knownPlayableCards/safeDiscards/visiblePlayableClues are computed with the SAME engine helpers (isPlayableCard at engine.ts:289-294, stack comparisons, isCriticalCard) that author the puzzle answers. (2) GameIQ scenarios set initialState = getFireworksPlayerView(scenario.state, actingPlayerId) with no sanitization (lib/benchmark/gameiq/fireworks.ts:36-39). (3) The certified runner serializes the ENTIRE view into the user prompt: `Initial state JSON: ${JSON.stringify(scenario.initialState)}` (lib/benchmark/gameiq/certified-runner.ts:251, invoked at :137). No stripping anywhere in the prompt path.

I verified the leak per category against the scenario authoring in lib/benchmark/fireworks/scenario-packs.ts:149-321, covering every GameIQ pack (basic/hard/memory per packs.ts:45-68):
- safe_play, combine_color_and_rank, endgame_play, old_clue_recall, negative_information, timing_inference: knowledge has color+rank and card is playable -> knownPlayableCards=[0]; expected action is `play cardIndex 0`. Direct transcription.
- safe_discard: stacks[color] >= rank -> safeDiscards=[0]; expected `discard cardIndex 0`. Direct.
- needed_clue, avoid_bad_play, critical_discard_avoidance: own-card knowledge lacks a proven-playable identity so knownPlayableCards may be empty, BUT the partner has a playable card so visiblePlayableClues contains exactly the expected clue action (e.g. clue_rank P2 rank 1). Direct.
Every scenario's top-weight expected action appears verbatim inside recommendations, so the model only needs to copy one entry — the benchmark measures JSON transcription, not Fireworks reasoning. The reviewer did NOT misread; there is no guard.

Severity: high, not critical. Blast radius is the GameIQ Fireworks scoring signal (3 packs), which it fully invalidates as a discriminator — exactly the subsystem's purpose. But it is benchmark-integrity, not a security/data-exposure or runtime-crash defect: no real user PII, no app breakage. It is live code (lib/benchmark/gameiq is the certified path, not in CLAUDE.md's legacy list), and the leak is already baked into the committed snapshot benchmarks/gameiq/v1/fireworks.json (recommendations present in all 20 serialized scenario states), confirming it ships to models.

**Suggested fix** — Strip recommendations from the view sent to models on the GameIQ/certified prompt path only. Cleanest: add a sanitized builder (e.g. getFireworksModelView or a `toCertifiedView(view)` that omits `recommendations`) and use it in lib/benchmark/gameiq/fireworks.ts:36 when constructing initialState, OR have gameIqScenarioPrompt drop `recommendations` before JSON.stringify. Do NOT remove recommendations from getFireworksPlayerView outright — it is consumed by the UI hint logic in components/games/fireworks/FireworksActionPanel.tsx:44,80 (knownPlayableCards/visiblePlayableClues). Note legalActions must remain in the scored scenario object because validateFireworksAction reads view.legalActions (lib/benchmark/gameiq/validation.ts:233); stripping legalActions is optional and only applies to the serialized prompt text, not the stored scenario. Also regenerate/scrub the committed benchmarks/gameiq/v1/fireworks.json snapshot so the leaked recommendations are removed there too.

### 5. trap-setup category has no authoring-time correctness check — only legality is verified

- **Severity:** High _(reviewer claimed medium; verifier adjusted)_ · **Confidence:** high · **Category:** scoring-validity
- **Subsystem:** GameIQ track
- **Location:** `lib/benchmark/gameiq/validation.ts:240-286`

**Evidence**

> validateScenarioCategory handles win-in-one, block-win, avoid-losing-move (Connect Four) and chess mate-in-one/legal-tactic, but trap-setup falls through validateConnectFourCategory's branches to `return ok()` (line 285). So the only guarantee for the 10 trap-setup scenarios is that expected.column is a legal column (validateConnectFourAction), not that it actually creates the advertised double threat ('creates two bottom-row threats').

**Impact** — Unlike win/block/avoid scenarios whose expected action is re-derived from the engine, trap-setup expected answers are unverified hand assertions. If a trap board were authored wrong (support stones mis-placed, target column not actually producing an open three), the test suite would still pass, and models would be graded against a column that doesn't achieve the stated objective. This is a latent data-integrity gap for a whole category.

**Verifier** — Confirmed and worse than the reviewer described. In lib/benchmark/gameiq/validation.ts, validateConnectFourCategory (lines 255-286) only special-cases win-in-one, block-win, and avoid-losing-move; trap-setup falls through to `return ok()` at line 285, so validateScenarioCategory (240-252) gives trap-setup zero correctness verification — only validateConnectFourAction legality (173-181) runs. The 10 trap-setup scenarios (1 base at connect-four.ts:88-115 + 9 generated at connect-four.ts:305-334) have hand-authored expected columns (`column: Number(target)` -> expected(...) at line 229), NOT engine-derived ones, exactly as claimed.

I empirically verified all 10 trap boards by dropping the expected disc and counting immediate winning follow-ups. 3 of 10 do NOT produce the advertised double threat (prompt says "creates two bottom-row threats"): trap-right-yellow (expectCol 5 -> 1 follow-up), trap-low-red (expectCol 0 -> 1), trap-high-yellow (expectCol 6 -> 1) — all edge-column traps where the open three runs off the board edge. Worse, for two of them a DIFFERENT column yields the real double threat: trap-low-red col 3 -> 2 follow-ups, trap-high-yellow col 5 -> 2. So the runtime scorer (runner.ts:93, actionMatchesExpected at validation.ts:321 — which compares model.column to the hand-authored expected column and never calls validateScenarioCategory) marks the genuinely correct double-threat move WRONG and the inferior single-threat move CORRECT. For trap-right-yellow no column achieves 2, so the scenario premise is impossible. This is a live benchmark feature (app/benchmark/page.tsx; gameiq is not in CLAUDE.md's legacy dead-module list), so the corrupted expected answers contaminate the GameIQ outcomeScore/moveQuality for real model runs (3/10 of the trap category). The missing validator is the only reason the test suite (scripts/test-gameiq-scenarios.mts:188) and corpus review (scripts/review-benchmark-corpus.mts:60) pass despite these defects. The reviewer slightly understated impact (called it a "latent data-integrity gap"); it is already active grading corruption, not merely latent.

**Suggested fix** — Add a trap-setup branch to validateConnectFourCategory: drop the expected disc via dropDisc(state, expected.column, 0), then count columns whose next drop produces a win for state.turn (reuse the getLegalColumns/dropDisc pattern from hasImmediateConnectFourWin) and require the count to be >= 2; fail otherwise so the test suite/corpus review surfaces the bad scenario. Running this immediately fails trap-right-yellow, trap-low-red, and trap-high-yellow. Then FIX the three defective boards' expected columns / blocker placement so they actually yield two winning follow-ups (e.g. trap-low-red should expect column 3, trap-high-yellow column 5; trap-right-yellow's board must be re-authored since no column gives a double threat). Optionally also assert the expected column is the maximal-threat column so suboptimal-but-double answers are caught.

### 6. 4-participant games (codenames) get zero wins and double-counted games per model

- **Severity:** High · **Confidence:** high · **Category:** scoring-validity
- **Subsystem:** Lab metrics / store / aggregate
- **Location:** `lib/benchmark/metrics.ts:867-901`

**Evidence**

> for (let i = 0; i < aiParticipants.length; i++) { const participant = aiParticipants[i]; ... const score = input.scoreFor(participant.modelId); score.games += 1; ... else if (winnerId === participant.id) score.wins += 1; else score.losses += 1;  — combined with winnerId = readString(result.winner) ("red"|"blue") and codenames participant ids "red-spymaster"/"red-operative"/"blue-spymaster"/"blue-operative" (lib/games/codenames/benchmark.ts:136,143-169).

**Impact** — For codenames the persisted winner is the team string ("red"/"blue") while participant ids are role-suffixed ("red-spymaster", etc.), so winnerId === participant.id is NEVER true. Every codenames model is therefore recorded as a loss (wins stays 0), making winRate / strategyScore systematically wrong for any model that played codenames. Worse, both red roles share the same modelId, so scoreFor(redModel) runs score.games += 1 twice per match (and blue twice), double-counting games, completions, legalActions, and schemaValid for the same model in a single match. Chess/connect-four/battleship are 2-participant with id-matching winners and are unaffected.

**Verifier** — Confirmed both halves of the finding by reading the live data flow.

ZERO WINS: codenamesMatchToGenericGameMatchRecord (lib/games/codenames/benchmark.ts:136) sets winner = record.result === "draw" ? null : record.result, and record.result is typed CodenamesTeam|"draw" = "red"|"blue"|"draw" (types.ts:3, benchmark.ts:31, resultForState at benchmark.ts:72). But participant ids are role-suffixed: "red-spymaster","red-operative","blue-spymaster","blue-operative" (benchmark.ts:143-169). In addGameMatch, winnerId = readString(result.winner) (metrics.ts:860) so winnerId is "red"/"blue", and the comparison winnerId === participant.id (metrics.ts:886) e.g. "red" === "red-spymaster" is NEVER true. With isDraw false this hits the else branch -> score.losses += 1 and wins stays 0 for every codenames model. Downstream: finalizeScore (metrics.ts:935) winRate = rate(wins, games - draws) = 0, and strategyScore = pctNumber(winRate ?? verifierPassRate ?? 0) = 0 (metrics.ts:950), feeding qualityScore at 30% weight. So codenames models are systematically penalized.

DOUBLE-COUNTING: scoreFor(modelId) (metrics.ts:245-254) returns a single shared MutableScore keyed by modelId. The loop iterates all 4 aiParticipants (metrics.ts:867), but the two red roles share record.redModel and the two blue roles share record.blueModel. So scoreFor(redModel) returns the same object twice per match -> score.games, completions, legalActions, schemaValid, schemaInvalid, fallbackActions, invalidActions all increment twice per model per match. perModelMoves/perModelInvalid are computed over 4 participants (metrics.ts:857-859), compounding the distortion. Also pollutes summary-card totals (metrics.ts:992-1003).

NOT legacy/dead code: contrary to a possible defense, this path is live. components/games/benchmark/codenames-runner.ts:52 saves the generic record; useBenchmarkDashboard.ts:122 reads getGenericGameMatchRecords() and passes them into buildBenchmarkDashboardData -> addGameMatch. lib/db/index.ts and the other modules CLAUDE.md flags as dead are not on this path.

ASYMMETRY confirmed: connect-four (participant ids "red"/"yellow", winner=record.result, benchmark.ts:62,70,79) and battleship (ids "blue"/"orange", benchmark.ts:140,147,154) use the participant id itself as the winner token AND have 2 distinct modelIds, so both id-matching and game counts are correct. Codenames is the sole broken converter.

Severity HIGH (not critical): the corruption is real and feeds visible leaderboard metrics (winRate, strategyScore, qualityScore, completions/legalActions totals), but it only triggers when a user actually runs codenames AI-vs-AI benchmark matches from the games UI, and it does not corrupt non-codenames models' own scores (only the shared summary totals). Blast radius is bounded to codenames participants plus global aggregate cards.

**Suggested fix** — Two independent fixes are needed in addGameMatch (lib/benchmark/metrics.ts:867-901):

1) Resolve team-level winner to participants before the win check. Derive the winning team from the winner token and credit a win to every participant whose id is on that team, rather than requiring winnerId === participant.id. Concretely, compute a per-participant isWinner: for 2-participant games keep id equality; for codenames, treat winnerId ("red"/"blue") as a team and set isWinner = participant.id startsWith winnerId + "-" (or, more robustly, normalize by matching the participant id's team prefix). The cleanest general solution: add an optional team field to GameParticipant and have each converter set it (codenames -> "red"/"blue"; connect-four/battleship -> the same value as id), then compare winnerId against participant.team falling back to participant.id for backward compatibility with already-persisted records.

2) De-duplicate per-model accumulation within a single match. Collapse participants by distinct modelId before incrementing games/completions/legalActions/schemaValid/etc., so a model occupying multiple seats counts once per match. Either aggregate the loop body per distinct modelId, or guard with a Set<string> of modelIds already counted for this match. Win/loss credit should then be per distinct modelId (a model wins the match if any of its seats are on the winning team). Keep per-model move/invalid distribution consistent (distribute over distinct models, not raw participant count) so legalActions/invalidActions are not inflated.

Because already-persisted codenames records carry winner="red"/"blue", any fix relying purely on a new participant.team field must also handle existing stored records (prefix match on the role-suffixed id covers them without a data migration).

### 7. Redaction only scrubs artifacts[].content; secrets/paths in traces, tool-calls, run events, verifier output and failures leak unredacted

- **Severity:** High _(reviewer claimed critical; verifier adjusted)_ · **Confidence:** high · **Category:** security-privacy
- **Subsystem:** Reports & redaction (export/import)
- **Location:** `lib/benchmark/redaction.ts:161-198`

**Evidence**

> redactBenchmarkBundle maps only `bundle.artifacts`: `const artifacts = bundle.artifacts.map((artifact) => { ... redactKnownSecretsWithCount(artifact.content) ... }); return { ...bundle, artifacts, redactionSummary: {...} };` — every other field is spread through untouched. ### 10. Run-level failure synthesizes per-(case x team) attempts that double-count traces with no caseId

- **Severity:** Medium _(reviewer claimed high; verifier adjusted)_ · **Confidence:** high · **Category:** data-integrity
- **Subsystem:** Certified run engine
- **Location:** `lib/benchmark/certified/run-engine.ts:133-174`

**Evidence**

**Verifier** — The double-counting CONCLUSION is real, but the reviewer's stated MECHANISM (case-less traces matching every case via `!trace.caseId`) is largely inaccurate.

Verified data flow (note the cited file path was wrong: it is lib/benchmark/model-call-traces.ts:80, not lib/benchmark/certified/model-call-traces.ts):

ALL four certified runners pass a truthy caseId: gameiq certified-runner.ts:142 and fireworks certified-runner.ts:472 pass input.context.caseIds[0]; toolreliability certified-runner.ts:97 and teamiq certified-runner.ts:196 pass benchmarkCase.id. validateCertifiedRunSelection (run-guards.ts:48-50) requires caseIds.length >= 1 and validates each is a real case. So certified traces effectively ALWAYS carry a caseId; the `!trace.caseId` branch at run-engine.ts:139 is dead for certified traces. The reviewer's examples are wrong: a budget reserveModelCall throw (model-call.ts:120) happens BEFORE the trace is created (line 171), so it produces no trace at all.

The REAL bug: the filter at run-engine.ts:136-140 scopes traces only by runId and caseId, never by team/participant. createFailedAttemptsForRunError runs in the run-engine.ts:72-88 catch block. Runners return their attempts only at the very end (model-runner.ts type is Promise<void|BenchmarkAttemptV2[]>, recorded post-resolve at run-engine.ts:71), but traces are persisted incrementally inside each model call (recordCertifiedModelCallTrace -> context.recordTrace, run-persistence.ts:87). So when a runner throws mid-way after some model calls, existingKeys (run-engine.ts:120-124) is empty and a synthesized failed attempt is created for EVERY (case x team) pair. Each persisted trace whose caseId === caseId then gets summed into that attempt for EVERY team M (no team scoping), so costUsd (line 159), inputTokens (160), outputTokens (164) and modelCalls: traces.length (168) are multiplied M-fold. Severity is medium, not high: it is confined to the failure synthesis path (status === 'failed' only), never affects successful-run leaderboard data, and the inflated attempts carry failure statuses. It is a data-integrity defect on the error path, easily triggered by any real run-level failure (budget/provider error) after >=1 model call with multiple teams or cases selected, but not a crash/security issue and not dead/legacy code (this is the live certified benchmark engine behind app/benchmark).

**Suggested fix** — Scope each trace to exactly one synthesized (case, team) attempt instead of letting it match every team (and, for gameiq/fireworks, every case). Concretely: (1) dedupe by trace id across the whole nested loop with a usedTraceIds Set so the same trace can never be summed into two attempts; and (2) attribute each trace to a single team — e.g. only the first teamCompositionId for that trace's case (input.context.teamCompositionIds[0]), recording zero cost/tokens/modelCalls for the same case under other teams. Keep the `!trace.caseId` fallback only as a safety net (assign such traces to the first case/team pair), but recognize it is effectively unreachable for certified traces. Note that participantId is not a reliable team key across tracks (gameiq/toolreliability set it to teamCompositionId, but teamiq uses `${team.id}:${role.slot}` and fireworks `fireworks:${playerId}`), so de-duping by trace id plus single-pair attribution is more robust than trying to filter traces by team. Add a regression test that fails a run after one model call with M>1 teams and asserts the summed modelCalls/costUsd across synthesized attempts equals the actual trace totals, not M times them.

**Verifier** — The mechanism is genuine. It stops only at the next bare `SEARCH` line or EOF, so any trailing content after a single op is folded into `replace`. This is asymmetric with extract.ts parseEditOps (lines 174-188), which tracks a `terminated` flag and pushes the op only `if (search.length > 0 && terminated)`, dropping un-terminated ops. The conflict-marker sibling parseConflictMarkerOps (572-578) likewise requires `>>>>>>> REPLACE`. The plain-keyword branch alone has no terminator. The downstream impact is real: evaluatePatchCase (runner.ts:257-263) feeds these ops to applyEditOps and compares with exact equality `applied.content === benchmarkCase.expectedContent`, so absorbed prose yields content_mismatch (classifyPatchFailure, 415) — a false-fail on the patch/firstAttempt metrics for a model that produced a correct edit.

However, the reviewer overstates blast radius and some triggers:

The reviewer's "second fenced block / trailing explanation" scenarios mostly do NOT trigger. The genuinely-triggering cases are narrower: the bare-keyword form emitted UNFENCED with prose after it, or prose inside the SAME fence after the REPLACE block.
(2) This branch only fires at all when extractArtifacts found no fenced ```edit``` block, no `<<<<<<< SEARCH` conflict-marker form, and no JSON patch (extractPatchForCase, 440-505), and the model used the literal bare `SEARCH`/`REPLACE` keyword form — an uncommon formatting that this fallback exists to rescue.

No real user file is read or overwritten; nothing is destroyed. The code is live (wired through lib/benchmark/toolreliability/index.ts to the benchmark UI), not legacy/dead per CLAUDE.md. The worst outcome is a model losing a patch-metric point on the leaderboard under a narrow output shape — a false negative, not data loss or a correctness bug in the product flow. That places true severity at low, not medium.

A blank line does not stop the replace loop (only `^SEARCH$`/EOF do), so even prose separated by a blank line is absorbed — the fix should not rely on blank-line detection alone.

**Suggested fix** — Mirror parseEditOps' terminated-op discipline: have parsePlainSearchReplaceOps require an explicit close for the replace block and discard the op if none is seen. A pragmatic, low-risk version: track a `terminated` flag set when a fence-close or a next-`SEARCH` boundary is hit, and additionally accept a sentinel such as an `END`/`REPLACE-END` line; if the replace block runs to EOF with no boundary, drop the op (don't fold trailing prose). Net: only push ops whose replacement region is bounded by a recognized terminator, matching extract.ts. This is a benchmark-fairness hardening, not a correctness-critical fix.

## Per-subsystem health & strengths

### Build leaderboard (`Build Lab` tab)

The original Build-mode worker leaderboard (benchmark page) renders accumulated per-model Build stats with sortable columns, a summary strip, and an expandable per-model detail. The core scoring is sound and correctly honors the project rule that "speed" is throughput (chars/s computed from responseMs/responseChars of successful responses only), never raw elapsed time; division-by-zero is guarded throughout, and infra denials are correctly kept out of quality. The most significant issue is a secondary-sort (tie-breaker) direction bug that flips with the primary sort direction, plus an availability column whose display ("-") contradicts how that column sorts. Aggregation math is otherwise consistent and the subsystem is broadly healthy.

**Strengths:**

- The leaderboard's Speed metric correctly implements throughput (charsPerSecond = responseChars/responseMs*1000) sourced only from successful, non-empty responses (responseMs/responseChars are accumulated solely on success in build-engine.ts:6149-6151), faithfully honoring the project rule that worker speed must be throughput, never raw elapsed time.
- Division-by-zero is guarded in every metric helper: qualityPerAttempt, approvalRate, availability, charsPerSecond, and judgeSummary.independentPct all check their denominator > 0 and return null, and the UI renders null as '-' via pct() and explicit null checks.
- Infra denials are cleanly separated from quality everywhere: qualityScore excludes unavailable, approvalRate's denominator excludes unavailable, and availability is tracked as its own axis — matching the documented intent that a free-tier 429 isn't the model's fault.
- normalizeStat() defensively backfills every field for records persisted before the field existed, so older stores don't crash the leaderboard or produce NaN — good forward/backward-compatibility hygiene.
- Judge independence is surfaced honestly: judgeSummary computes independentPct from approve/fix verdicts only, the detail view flags self-graded share as 'lower trust', and accumulateModelStats correctly omits a judge that contributed zero verdicts.
- Sort handling for null values is explicit and consistent (nulls always sort to the bottom regardless of direction via the va==null/vb==null checks), and string vs numeric comparison is branched correctly with localeCompare for the model-name column.

### Lab metrics / store / aggregate

The core data layer aggregates two parallel benchmark surfaces: a legacy "lab" dashboard (buildBenchmarkDashboardData) over game matches / build stats / metric values, and a newer "certified" dashboard (buildCertifiedBenchmarkDashboardData + scoring/aggregate.ts) over V2 attempts. Both are live (consumed by components/benchmark/useBenchmarkDashboard.ts). The certified scoring path (aggregate/types/pareto) is well-built with consistent average-vs-sum discipline, finite-guards, and stable sorts. The legacy game-match aggregation, however, has several correctness bugs: win attribution and game-count double-counting are wrong for 4-participant games (codenames), the per-day quality trend mixes a per-participant numerator with a per-match denominator, and verifierPassRate can be corrupted by an arbitrary "quality" metric value. Persistence (store.ts merge/dedup/cascade) is solid; the main risks are statistical validity in the legacy dashboard and small-sample averages presented without sample-size guards.

**Strengths:**

- Certified scoring (scoring/aggregate.ts) is disciplined about average-vs-sum: it keeps separate sum/sample pairs (costSamples, durationSamples, toolReliabilitySamples) and only averages over present samples, so a missing costUsd does not silently count as $0 in the mean.
- NaN/Infinity handling is consistent and centralized: finiteOrNull/finiteMetric/isFiniteNumber gate every numeric path, round() returns 0 for non-finite, and pareto comparableValue maps non-finite to the correct sentinel (-Inf for 'higher', +Inf for 'lower').
- store.ts merge/import is robust: mergeById/mergeByKey are last-write-wins by stable key, buildCheckpoints correctly keyed by discussionId (matching the one-checkpoint-per-discussion store invariant), and a thorough validateBenchmarkReportBundleV2 deep-validates every record (including JSON-object-string fields) before import, with a stable FNV-1a bundle hash that normalizes key order and excludes bundleHash itself.
- Cascade deletes (deleteBenchmarkAttemptsCascade/deleteBenchmarkRunCascade) carefully gather transitively-referenced artifact ids from both the attempt and its verifier results before deleting, and persistBenchmarkRunFile garbage-collects run blobs that no longer have evidence.
- rankBy* helpers return sorted copies ([...rows]) rather than mutating in place, and use deterministic multi-key tiebreakers (score, then pass rate, then attempts, then displayName) so leaderboard ordering is stable.
- Pareto frontier dominance is implemented correctly with a strict-improvement requirement and self-exclusion, and handles the all-equal and missing-dimension cases sanely.

### Certified run engine

The certified-run orchestration subsystem is generally well-structured: it cleanly separates run lifecycle (run-engine), budget enforcement (budget/run-budget), persistence (run-context/run-persistence), and classification (classify-*). The in-memory run context snapshots all records synchronously, and the store's upsert-then-flush model means dashboard rebuilds read fresh data (no stale-read race). The most serious issue is a scoring-validity inconsistency: the `invalid_case` status is treated as "invalid / excluded from scoring" by the failures module, the reports module, and the attempt-detail UI, but the actual leaderboard gate `isScoredCertifiedAttempt` forgets it and counts broken-fixture attempts against the model. Secondary issues involve cost/token double-attribution on the run-level error path, an unimplemented user-abort path, soft-only mid-call budget enforcement, and a timeout misclassification.

**Strengths:**

- Run context is an in-memory Map-backed snapshot with synchronous upsert + flush in the store, so rebuildCertifiedDashboardData reads fresh data — there is no persistence stale-read race between recording and dashboard rebuild.
- Budget off-by-one is correct: reserveModelCall checks before and after incrementing modelCalls, so exactly maxModelCalls calls are permitted and the (N+1)th is rejected (budget.ts:98-112).
- Error classification is layered and mostly consistent (classify-run-result + classify-failure + failures.ts), cleanly separating model-accountable failures from invalid/excluded runs, and the source/severity mapping in model-runner.ts is sensible.
- callCertifiedModel always records a trace on both success and failure paths (model-call.ts:171-205 and 236-273), preserving partial rawResponse and a finalStatus of parse_error/provider_error, which is excellent for post-hoc debugging and reproducibility.
- The harness certification gate (assertHarnessCertificationCanRun) plus mock-model self-tests (oracle/bad-json/forbidden-tool/wrong-patch/timeout) give a real, deterministic pre-flight that a run cannot start unless the harness classifies known-bad behaviors correctly.
- Defensive budget input sanitization via positiveInteger (rounds, clamps to >=0, rejects NaN/Infinity) prevents negative or non-finite usage from corrupting the running budget totals.
- The per-call timeout wrapper (withCertifiedModelCallTimeout) properly closes the underlying async iterator in a finally block and clears the timer, avoiding leaked streams/timeouts.

### GameIQ track

GameIQ drives single-model game-move benchmarks across Connect Four, Chess, Battleship, Codenames, and Fireworks. The validation layer (validation.ts) correctly delegates legality to the real game engines, and authoring-time category validators (win-in-one, block-win, avoid-losing-move, mate-in-one, legal-tactic) genuinely re-derive correctness from the engines — all 175 shipped scenarios pass scripts/test-gameiq-scenarios.mts. The core weakness is scoring validity: the "correct" signal is mostly exact-match against a single hand-authored expected action, the chess (and several other) packs are dominated by exact duplicates of 2-4 base positions, and the data benchmarks/gameiq/v1/*.json manifests are stale and diverge from the live TypeScript source of truth. These don't crash anything but materially distort leaderboard scores and the published manifest record.

**Strengths:**

- Legality is delegated to the real game engines rather than re-implemented: validation.ts calls getLegalColumns/dropDisc, isLegalMove/fromFEN/makeMove, isLegalBattleshipTarget, validateCodenamesClue/submitCodenamesGuess, and fireworksActionsEqual — so 'legal' truly means legal per the canonical rules.
- Authoring-time category validators genuinely re-derive correctness from the engine for the unique-solution categories: win-in-one re-runs the drop and checks status==='win' && winner===turn; block-win/avoid-losing-move confirm an immediate opponent threat exists and that the expected move removes it; mate-in-one checks makeMove yields 'checkmate'. This catches mis-authored scenarios, and all 175 shipped scenarios pass scripts/test-gameiq-scenarios.mts.
- Structured-output handling is robust: isStructuredGameIqAction tolerates the structured-output null promotion (chess) and the run separates 'structured shape' from 'legal' from 'correct', so a malformed response degrades gracefully (fallbackUsed) rather than throwing, and the runner wraps provider errors into a fallback result instead of aborting the whole attempt.
- Codenames clue scoring correctly uses legality-as-correctness (actionMatchesExpected special-cases clue-selection) and case-insensitive word comparison, which is the right model for an open-ended clue task.
- Pack accessors defensively deep-clone (clone() via JSON round-trip) so callers cannot mutate the shared canonical scenario data, and digests are computed with a stable key-sorted stringify so ordering changes don't perturb the hash.

### TeamIQ track

TeamIQ enumerates multi-model team compositions, runs them through the ToolReliability certified harness, computes "team lift" vs solo baselines, ranks combos on a Pareto frontier, and surfaces recommendation cards. The composition/hashing layer is solid and well-tested, and the Pareto core and baseline-linking are carefully guarded (track/harness/scoring-version matching, complete-baseline requirement). However several scoring-validity issues undermine the headline metric: the cost/speed-adjusted lift scales the wrong way for negative lift, the Pareto frontier double-counts score via a teamLift dimension that structurally rewards weak member baselines, the "wasteful" classification ignores cost whenever lift is positive, and the certified team "answer" is a single role's verbatim output (no synthesis) while the attempt is charged the full N-model cost — so team-vs-solo comparisons are systematically unfavorable to teams and "lift" is near-zero by construction for degenerate same-model teams the UI lets users build.

**Strengths:**

- Composition identity is robust: normalizeTeamRoles sorts roles deterministically and comboHashFor/stableStringify produce a stable, order-independent hash (verified by the round-trip test in scripts/test-teamiq-baselines.mts), so the same team always groups together regardless of selection order.
- Baseline linking is conservative and correct: linkTeamLiftBaselines only emits a lift when a complete solo baseline exists for every unique member AND matches on caseId, track, harnessVersion, and scoringVersion (baselines.ts:93-100), preventing apples-to-oranges lift comparisons across harness/scoring changes.
- The Pareto core (scoring/pareto.ts) is a clean, correct strict-domination implementation with proper handling of non-finite values via comparableValue mapping to +/-Infinity by direction, and short-circuits empty dimension lists.
- Combinatorial blowup is bounded on the live path: createTeamIqToolBenchCompositionsFromSelection emits a fixed 5 strategies, maxModelsForStrategy caps role counts, and planTeamIqExperiment (test-only) honors an optional maxCombos with a labeled break to stop enumeration.
- Slot naming is deliberately designed to survive role sorting (e.g. '99-strong-judge', '01-architect') so finalOutputForTeam's findIndex on the sorted roles array still aligns with the roleOutputs built in the same sorted order.
- Good defensive numeric hygiene throughout: finiteNumber/finiteOrNull/round guard against NaN/Infinity, and recommendation maxBy/minBy skip non-finite candidates rather than propagating them into rankings.

### Tool Reliability track

The Tool Reliability track is a deterministic, well-factored verifier suite: it generates 125 canonical cases plus a 70-case stress pack, applies SEARCH/REPLACE and JSON patches through the shared artifact extractor, and aggregates per-metric pass rates into a weighted score. Schema conformance, patch application, and large-file "no whole-file rewrite" detection are mostly sound and covered by tests. The most serious gap is the command-safety classifier used by the forbidden-action track: an unanchored allow-pattern combined with a chaining-blind classifier lets clearly unsafe chained commands score as fully safe. Several scoring-validity issues (free credit for absent metrics, a non-content-gated reported "no rewrite" rate, duration encoded as attempt count) also undermine cross-pack comparability.

**Strengths:**

- Patch evaluation reuses the production extract.ts parser (applyEditOps/extractArtifacts) and adds tolerant fallbacks (fenced edits, pathless SEARCH/REPLACE, JSON patch objects) with explicit path-mismatch handling and a clear failure taxonomy (unsupported_patch_format / patch_did_not_apply / content_mismatch), so the verifier classifies failures the way a human reviewer would.
- The large-file stress evaluator gates on exactContentMatch AND minimal-patch budgets AND a whole-file-rewrite guard simultaneously, and the test suite (test-toolreliability-stress-cases.mts) proves a whole-file rewrite that produces identical final content is still rejected — a genuinely hard property to get right.
- Schema validation is strict and total: it rejects non-JSON, non-objects, wrong types, out-of-range numbers, enum violations, and under-length string arrays, and the certified runner mirrors the internal schema into a real JSON Schema for structured output so the model is told exactly what is expected.
- Case generation is fully deterministic with namespaced ids and canaries, and validateToolReliabilityCasePack enforces id namespacing, canary prefixes, duplicate detection, category coverage, and full metric coverage — making accidental pack regressions caught at test time.
- The diagnostics layer assigns failure accountability (provider / aiboard / test_design / model) from event evidence, which cleanly separates 'model got it wrong' from 'harness/fixture problem' for the dashboard.

### WorkBench track

WorkBench runs a model build against an inline fixture workspace on a loopback-only bench-runner, then scores a deterministic `verifier.mjs` (snippet/diff/syntax assertions) into verifiedQuality / efficiency. The executor, case-loader, and runner sandboxing (path traversal, command allowlist, loopback binding, CORS origin allowlist) are carefully written and well-tested. The most serious problems are scoring-validity and gameability: the verifier reads its own pass/fail criteria from a workspace file the model can freely overwrite, the tool-reliability dimension is hardcoded to 1.0 and never persisted, and several "surgical" / "preserves unrelated code" assertions are substring checks that don't actually validate what they claim.

**Strengths:**

- Strong path-traversal defenses: resolveSafePath/resolveSafeChild in bench-runner.mjs reject absolute paths, drive letters, and '..' segments and re-check the resolved path stays under the attempt root; case-loader.assertSafeRelativePath mirrors this for verifier.resultFile.
- Runner is loopback-only by construction (isLoopbackHost refuses non-loopback binds), token-gated on every request, body-size capped, output-capped, and uses a strict CORS origin allowlist rather than reflecting arbitrary origins.
- Command execution is gated by an exact-match allowlist (assertAllowedCommand) derived only from setup/verifier/explicit allowed commands, and the loader forbids network:'none' when commands must run, surfacing the enforcement gap honestly.
- case-loader performs thorough, well-typed manifest validation (required strings, positive numbers, enum membership, schemaVersion pinning) with clear error messages, and createWorkBenchCaseHash uses a stable, key-sorted JSON serialization for deterministic hashing.
- The verifier parser (verifier.ts) is defensive: balanced-brace JSON extraction, weight-based scoring with sane fallbacks (clamp01, total-weight guards), strict boolean/number type checks, and a normalized rawJson echo.
- Failure handling is structured and exhaustive: classifyPrepare/Build failures map to specific CertifiedAttemptStatus codes, and createFailedWorkBenchAttempt always emits a synthetic verifier result + log artifact so failed attempts remain inspectable and consistently zero-scored.
- The TS test suite cross-checks every challenge's reference solution (pass) and negative control (fail) and accepts equivalent non-reference formatting, which is a good validity discipline even though it runs against the TS verifier rather than the deployed one.

### Fireworks benchmark

The Fireworks subsystem programmatically generates tactics/memory scenario packs and full-game cases, runs them through a certified TeamIQ runner with deterministic fallbacks and calibration bots, and exposes the same scenarios as GameIQ single-model cases. The engine, scoring, and wiring are coherent and well-typed, but there are several serious scoring-validity issues: the player view that is serialized into the model prompt embeds a precomputed optimal-move recommendation list (answer leakage), and when a model emits invalid output the strong deterministic fallback is what gets scored — inflating headline quality. Memory scenarios fabricate prior clue events that then contaminate aggregate clue/useful-clue metrics, and the "random" calibration baseline is actually a single deterministic trajectory. The v0.1 JSON files are documentation-only manifests (not loaded), so version drift is cosmetic.

**Strengths:**

- Scenario and case ids/digests are deterministic and stable (stableFireworksScenarioPackDigest uses a sorted-key stableStringify + FNV hash), giving reproducible pack identity and good contamination metadata (canary, referenceSolutionPrivate).
- Scenario-pack counts are internally consistent and match the v0.1 manifests (6 tactics categories x10 = 60, 4 memory x10 = 40, 2 player counts x10 = 20).
- forbiddenActions are checked before expectedActions in scoreFireworksScenarioAction, so trap actions correctly score 0 even though they are legal, which is the right precedence for avoid_bad_play / critical_discard_avoidance.
- Failure taxonomy is thorough and well-categorized (provider vs parser vs rules; illegal_clue vs illegal_action vs invalid_json), with stable per-case failure ids and sequence numbers.
- Team-lift baseline linking is careful: it matches solo attempts to team members on caseId/track/harnessVersion/scoringVersion and only links when every member has a solo baseline, avoiding cross-version comparisons.
- The calibration bot set spans a sensible competence ladder (random, always-discard, safe-clue, greedy-playable, forgetful) and reuses the same engine + scoring path as real attempts, so baselines are apples-to-apples in mechanics.

### Reports & redaction (export/import)

This subsystem exports a V2 benchmark bundle as JSON/Markdown, redacts secrets/PII before export, and re-imports bundles via id-keyed merge. The redaction patterns themselves (API keys, runner tokens, SSH keys, env-var secrets, absolute local paths) are well-constructed and tested for the artifact channel. However, redaction is applied to only ONE of many free-text channels in the bundle: secrets and local paths in model-call traces, tool-call traces, run events, verifier stdout/stderr, and failure messages pass through to the exported file completely unredacted. Import validation is structurally thorough for typed sub-records but does not validate the `content`/free-text fields it later feeds to the regex redactor, creating a crash-on-re-export vector; the integrity `bundleHash` is never verified on import; and the id-keyed merge silently overwrites local records on collision.

**Strengths:**

- The secret-detection regexes are thoughtfully scoped and ordered: anthropic keys before openai (negative lookahead `sk-(?!ant-)`), distinct runner-token forms, both header-style and JSON-style authorization values, and SSH key blocks with the END marker optional so a truncated header is still caught.
- Redaction runs on the export path before download (exportBenchmarkReportBundleV2 → redactBenchmarkBundle), and the Markdown report is built from the already-redacted bundle, so the JSON and Markdown share the same scrubbed source for the channels that are covered.
- The replacement-callback group handling in redactKnownSecretsWithCount (`redactor.replacement(args[0], ...args.slice(1, -2))`) correctly strips the trailing offset/string args so capture-group prefixes/suffixes are preserved (e.g. Authorization header prefix is kept while the token is redacted).
- previewSecret only ever exposes 4 leading + 4 trailing chars and that preview is returned from scanArtifactForSecrets but is NOT persisted into the exported redactionSummary (only `kind` goes into warnings), so the preview itself is not a leak vector in exports.
- Import error handling is wired end-to-end: importJson(file).catch(reportError) in BenchmarkLab.tsx surfaces JSON.parse and validation failures to the user instead of failing silently, and the file input resets its value so re-selecting the same file re-triggers import.
- Import is a non-destructive merge (mergeById/mergeByKey) rather than a wholesale replace, so a malformed import cannot wipe the existing store, and the structured per-record validators (validateBenchmarkCaseV2, etc.) are genuinely thorough for the typed sub-records.

### UI — charts & panels

The benchmark UI is a large, well-structured set of presentational React components (Recharts charts, tables, scorecards, and certified/teamiq/workbench/fireworks panels) backed by lib/benchmark/metrics.ts. It has solid empty/locked/loading states, defensive parsing of the untyped certified dashboard, and confirmation guards on destructive deletes. The most material problems are (1) a statistical dilution bug in the Performance-Over-Time quality line, (2) cross-component inconsistency in how the same verifiedQuality metric is scaled/labeled, (3) charts with no accessible text alternative or keyboard affordance, and (4) several color-only encodings. Numbers are generally correct, but presentation consistency and accessibility need attention.

**Strengths:**

- Robust state handling: locked (passphrase), loading, and empty/zero-data states are consistently rendered (BenchmarkLab, BenchmarkModelScorecards, every chart via EmptyChart, CertifiedEmptyState/CertifiedTrackEmptyState), so the UI degrades gracefully with no data.
- The certified dashboard is treated as untyped (certified: unknown) and parsed defensively with readRecord/readArray/readNumber/readString guards and fallbacks, which tolerates imported bundles with missing or differently-named fields without crashing.
- Destructive actions (delete attempt, delete provider-error batch, reset Build stats) are all guarded by window.confirm with in-flight locking (deletionInFlightRef) and clear success/error messaging.
- Scoring discipline matches project conventions: provider/harness/environment/user-aborted attempts are explicitly excluded from leaderboard math (isScoredCertifiedAttempt), and Build quality never counts provider denials — surfaced honestly in UI copy and the FailureTaxonomyPanel.
- Tables consistently use overflow-x-auto with min-w wrappers and tabular-nums for numeric alignment, and most pickers/buttons have sensible aria-labels (e.g. BuildModelRow expand/reset buttons, WorkBenchRunnerStatus labeled inputs).
- Shared, single-purpose formatters (format.ts pct/usd/duration/shortModel) keep most number/currency/time formatting consistent across the panels that use them.
